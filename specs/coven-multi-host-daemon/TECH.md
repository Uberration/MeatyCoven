# Coven Multi-Host Daemon Architecture - TECH

**Status:** Draft v0.1 - 2026-07-04
**Owner:** Coven runtime - Coven Cave
**Tracks:** GitHub issues #264, #265, #267, #268, #269, #270

## Design constraints

- The hub is the only canonical writer for familiar memory, loop state, node registry, and global/per-executor queues.
- Executor nodes are dispatched by the hub; they do not initiate authority-bearing contact.
- Laptop travel clients run a local-only sub-daemon and reconcile through the hub.
- Offline deltas append results and proposed memory changes; they never overwrite canonical memory directly.
- The existing same-user Unix socket remains the local control plane per machine.
- Remote links are explicit hub-managed transports, not raw socket exposure.

## State model

### Hub tables

The exact schema can evolve, but the hub needs these durable entities:

| Entity | Purpose |
| --- | --- |
| `node_registry` | Known nodes, role, transport, capabilities, last health, and availability. |
| `node_capabilities` | Normalized capability facts used by scheduler decisions. |
| `job_queue` | Global job queue with required capabilities, priority, and current assignment. |
| `executor_queue` | Per-node subqueue preserving assigned work if a node disappears. |
| `loop_state` | Durable loop progress, pause/resume markers, and dispatch history. |
| `travel_profiles` | Generated profile metadata, source revisions, expiry, and revocation state. |
| `travel_deltas` | Uploaded offline delta envelopes and reconciliation status. |
| `scheduler_decisions` | Explanation records for Cave and debugging. |

### Laptop travel state

The laptop sub-daemon stores:

| Entity | Purpose |
| --- | --- |
| `travel_profile_cache` | Read-only profile blobs and metadata. |
| `offline_queue` | Local work accepted while disconnected. |
| `offline_events` | Local session events awaiting upload. |
| `offline_artifacts` | Result files and metadata awaiting upload. |
| `handoff_state` | Current reconnect state and sync progress. |

Laptop state is disposable from the hub's perspective. Losing it can lose offline work, but it must not corrupt hub state.

## Travel profile format

Travel profiles are generated by the hub and consumed by laptop sub-daemons.

```json
{
  "version": "0.1",
  "profile_id": "travel_01J...",
  "generated_at": "2026-07-04T00:00:00Z",
  "expires_at": "2026-07-11T00:00:00Z",
  "stale_after": "2026-07-06T00:00:00Z",
  "source_hub": {
    "hub_id": "hub_01J...",
    "display_name": "Val's server",
    "public_key_fingerprint": "SHA256:..."
  },
  "scope": {
    "principal": "@BunsDev",
    "familiar": "sage",
    "workspace_id": "workspace_01J..."
  },
  "source_revision": {
    "memory_revision": "mem_01J...",
    "loop_revision": "loop_01J..."
  },
  "permissions": {
    "mode": "travel-read-only",
    "allowed_local_agents": ["lightweight"],
    "allow_memory_overwrite": false,
    "allow_heavyweight_local_work": false
  },
  "payload": {
    "memory_context": [],
    "workspace_context": [],
    "policy_context": []
  }
}
```

Transport/storage requirements:

- Serialize as canonical JSON or MessagePack before compression.
- Compress with a deterministic supported codec.
- Store a content hash beside the blob.
- Mark profiles read-only on disk after write where the platform supports it.
- Refuse use after `expires_at`.
- Warn after `stale_after`.
- Refuse use if the source hub identity is revoked or unknown.

### Upload offline delta

`POST /api/v1/travel/deltas`

Request:

```json
{
  "profileId": "travel_01J...",
  "sourceHubId": "hub_01J...",
  "sourceRevision": {
    "memoryRevision": "mem_01J...",
    "loopRevision": "loop_01J..."
  },
  "clientId": "laptop_01J...",
  "offlineStartedAt": "2026-07-04T12:00:00Z",
  "offlineEndedAt": "2026-07-04T18:00:00Z",
  "events": [],
  "artifacts": [],
  "proposedMemoryAdditions": []
}
```

Response:

```json
{
  "deltaId": "delta_01J...",
  "state": "hub_resumed",
  "acceptedEvents": 12,
  "acceptedArtifacts": 3,
  "memoryReviewState": "queued",
  "hubRevision": {
    "memoryRevision": "mem_01K...",
    "loopRevision": "loop_01K..."
  }
}
```

Server behavior:

- Reject unknown, expired, or revoked profile IDs.
- Reject source hub mismatches.
- Store the raw delta envelope before reconciliation.
- Append accepted events/artifacts.
- Queue proposed memory additions for hub-side review.
- Never apply proposed memory additions as direct overwrites.
- Return the reconnect state needed by Cave.

Acceptance coverage for #268:

- hub accepts offline delta uploads;
- reconciliation appends offline results;
- canonical familiar memory cannot be overwritten;
- reconnect state can expose `handoff_pending`, `syncing_delta`, and `hub_resumed`.

## Executor protocol and SSH dispatcher

The stateless executor protocol is versioned as `coven.executor.v1` and is
shared by stationary and compute executor roles. The hub is the only side
that initiates contact:

- The hub polls availability by running `coven executor probe` on the node
  over an outbound transport. Executors never push registration or
  heartbeats to the hub.
- The hub dispatches work by running `coven executor run-job` and sending
  the job spec on stdin.
- Transports are hub-owned: `ssh` (batch mode, `StrictHostKeyChecking=yes`,
  outbound only) or `local` (private-network/same-host process launch).
  The raw daemon socket is never exposed.

### Probe envelope (`coven executor probe`)

```json
{
  "protocolVersion": "coven.executor.v1",
  "role": "compute_executor",
  "capabilities": ["shell", "gpu"],
  "available": true,
  "queuePressure": 0,
  "covenVersion": "0.0.0",
  "probedAt": "2026-07-06T00:00:00Z"
}
```

Both executor roles share this envelope; they differ only in the role and
capabilities they advertise (configured in `<covenHome>/executor.json`).

### Job spec (stdin of `coven executor run-job`)

```json
{
  "protocolVersion": "coven.executor.v1",
  "jobId": "job_01J...",
  "hubId": "hub_01J...",
  "requiredCapabilities": ["gpu"],
  "command": ["sh", "-c", "…"],
  "cwd": "/work/checkout",
  "env": {"KEY": "value"},
  "stdin": "optional payload",
  "timeoutSeconds": 300,
  "context": {"workspaceId": "workspace_01J..."}
}
```

The job spec carries everything the executor needs (argv, cwd, env, stdin
payload, and opaque hub-provided context), so the node runs it without
local durable authority. `run-job` reads no hub-authoritative state.

### Result envelope (stdout of `coven executor run-job`)

```json
{
  "protocolVersion": "coven.executor.v1",
  "jobId": "job_01J...",
  "status": "completed",
  "exitCode": 0,
  "stdout": "…",
  "stderr": "…",
  "startedAt": "2026-07-06T00:00:00Z",
  "finishedAt": "2026-07-06T00:00:05Z",
  "durationMs": 5000,
  "error": null
}
```

`status` is one of `completed`, `failed`, `timeout`, `rejected`, or
`transport_error` (the last is synthesized by the hub-side dispatcher when
the node is unreachable or replies with a malformed envelope, so callers
always see one shape).

### Hub routes

- `GET /api/v1/hub/nodes` — registry with capability metadata and last-known
  availability.
- `POST /api/v1/hub/nodes` — operator registers/updates a node (role,
  capabilities, and the structured `transportConfig` dispatch link).
  Live availability is earned through hub-initiated polls.
- `POST /api/v1/hub/nodes/:nodeId/poll` — hub polls the executor outbound and
  records capabilities, availability, and last error; availability
  transitions hold/resume the node's persistent subqueue.
- `POST /api/v1/hub/nodes/:nodeId/dispatch` — hub dispatches a job outbound,
  persists the dispatch record plus normalized envelope, treats the
  attempt as an availability observation, and advances a matching
  hub-queue job from the envelope.
- `GET /api/v1/hub/dispatches/:jobId` — durable dispatch record (job spec,
  envelope, status).

Acceptance coverage for #267:

- hub dispatches jobs outbound over SSH/private network;
- hub polls executor availability; executors never push to the hub;
- job specs carry full context so executor nodes need no local durable
  authority;
- executors return stdout/stderr/result metadata in a normalized envelope;
- the hub records executor capability metadata and last-known
  availability; and
- stationary and compute roles share the base protocol while advertising
  different capabilities.

## Scheduler

Scheduler inputs:

- job required capabilities;
- familiar policy;
- node role;
- node availability;
- node queue pressure;
- node health age;
- loop resumability;
- hub reachability;
- travel state;
- laptop battery/power source where available; and
- user override policy.

Scheduler output:

```json
{
  "decisionId": "sched_01J...",
  "jobId": "job_01J...",
  "target": {
    "role": "compute_executor",
    "nodeId": "node_01J..."
  },
  "reason": "compute executor has required gpu capability and lowest queue pressure",
  "inputs": {
    "requiredCapabilities": ["gpu", "long-running-loop"],
    "queuePressure": "low",
    "travelState": "hub_active"
  }
}
```

### Redispatch or pause loop

`POST /api/v1/scheduler/redispatch`

This route is the deterministic failure-simulation seam for executor loss. The hub preserves the failed executor's subqueue, then either redispatches a resumable loop to the best matching available executor or pauses the loop when no safe alternate exists.

Request:

```json
{
  "loopId": "loop_01J...",
  "jobId": "job_01J...",
  "currentNodeId": "node_primary",
  "requiredCapabilities": ["gpu", "long-running-loop"],
  "loopResumable": true,
  "nodes": [
    {
      "nodeId": "node_primary",
      "role": "compute_executor",
      "available": false,
      "capabilities": ["gpu", "long-running-loop"],
      "queuePressure": 3,
      "queuedJobIds": ["job_01J..."]
    },
    {
      "nodeId": "node_fallback",
      "role": "compute_executor",
      "available": true,
      "capabilities": ["gpu", "long-running-loop"],
      "queuePressure": 1,
      "queuedJobIds": []
    }
  ]
}
```

Response:

```json
{
  "decisionId": "sched_01J...",
  "state": "redispatched",
  "loopId": "loop_01J...",
  "jobId": "job_01J...",
  "target": {
    "role": "compute_executor",
    "nodeId": "node_fallback"
  },
  "reason": "node_primary went offline; redispatched resumable loop to node_fallback",
  "preservedSubqueue": {
    "nodeId": "node_primary",
    "jobIds": ["job_01J..."]
  },
  "nodeAvailability": [
    {
      "nodeId": "node_primary",
      "role": "compute_executor",
      "available": false,
      "queuePressure": "medium"
    }
  ],
  "createdAt": "2026-07-04T00:00:00Z"
}
```

The resulting state is durable and queryable by loop id:

`GET /api/v1/scheduler/loops/:loopId`

Response:

```json
{
  "decisionId": "sched_01J...",
  "state": "redispatched",
  "loopId": "loop_01J...",
  "jobId": "job_01J...",
  "target": {
    "role": "compute_executor",
    "nodeId": "node_fallback"
  },
  "reason": "node_primary went offline; redispatched resumable loop to node_fallback",
  "preservedSubqueue": {
    "nodeId": "node_primary",
    "jobIds": ["job_01J..."]
  },
  "nodeAvailability": [
    {
      "nodeId": "node_primary",
      "role": "compute_executor",
      "available": false,
      "queuePressure": "medium"
    }
  ],
  "createdAt": "2026-07-04T00:00:00Z",
  "updatedAt": "2026-07-04T00:00:00Z"
}
```

Rules:

- Prefer hub/executor execution for heavyweight work.
- Avoid heavyweight laptop-local work in `travel_local` and `travel_stale`.
- Pause or redispatch long-running loops when executor health becomes stale.
- Preserve per-executor queue entries when an executor disappears until the hub explicitly redispatches or cancels.
- Record every non-trivial scheduling decision for Cave/debugging.

Acceptance coverage for #269:

- executor chosen by capability, availability, and queue pressure;
- long-running loops can pause and redispatch;
- laptop-local agents are battery/travel aware;
- reason/debug data is exposed.

## Failure simulations

The release gate for #270 should simulate these scenarios with deterministic fixtures:

1. Laptop loses hub connectivity and enters `travel_local`.
2. Laptop reconnects, enters `handoff_pending`, transitions through `syncing_delta`, and ends in `hub_resumed`.
3. Compute executor goes offline mid-loop; hub preserves queue state and pauses or redispatches.
4. Stationary executor goes offline; hub preserves subqueue and reports availability accurately.
5. Server hub crashes and restarts from persistent state.
6. Travel client attempts canonical memory overwrite; hub rejects overwrite and stores it only as a proposed addition/review item.
7. Stale travel profile crosses `stale_after`; Cave-visible state becomes `travel_stale`.
8. Expired travel profile crosses `expires_at`; laptop refuses travel-mode execution for that profile.

## Cave contract

Cave should never infer multi-host authority from generic daemon health alone.

It needs explicit fields for:

- current role;
- active hub identity;
- hub reachability;
- travel profile freshness;
- reconnect state;
- pending delta size/count;
- sync progress;
- scheduler target;
- scheduler explanation; and
- node availability.

Cave may cache display state, but Coven owns state transitions.

## Security and auth notes

Multi-host mode requires a separate transport/auth design for hub-to-node links.

Hard requirements:

- Do not expose `<covenHome>/coven.sock` over TCP.
- Do not let a browser page talk directly to a daemon socket.
- Authenticate hub-to-node dispatch.
- Pin or verify node identity.
- Treat executor results as untrusted until the hub validates job identity and expected outputs.
- Keep provider credentials in provider/harness auth flows.
- Keep travel profiles scoped and expiring.
- Log reconciliation outcomes without logging secrets.
